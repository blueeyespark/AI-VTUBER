# Memory and Knowledge Model

## Memory classes

- **Session memory:** temporary working context.
- **User memory:** preferences and facts explicitly approved for retention.
- **Project memory:** decisions, files, tasks, failures, and lessons for a project.
- **Procedural memory:** approved workflows and tool-use patterns.
- **Relationship context:** consent-aware interaction history, never a permanent judgment of worth.
- **System memory:** configuration, module history, migrations, and security events.
- **Legacy memory:** protected founding and stewardship records.

## Knowledge classes

- **Verified knowledge:** supported by reliable evidence and provenance.
- **Reported knowledge:** attributed to a source but not independently verified.
- **Hypothesis:** testable explanation requiring validation.
- **Idea:** creative concept not presented as fact.
- **Simulation result:** output bounded by stated assumptions.
- **Personal memory:** user- or project-specific information with access controls.

## Required operations

Every retained record needs an identifier, timestamps, provenance, confidence or evidence status, owner, sensitivity class, retention rule, access policy, and version history.

Users and stewards need tools to:

- inspect why a memory exists;
- correct or annotate it;
- export it;
- delete it when policy permits;
- restrict its use;
- trace which action relied on it.

## Learning rule

Blue learns from approved sources, feedback, documents, experiments, and tool results. It does not silently convert every interaction into permanent memory or retrain itself without review.
